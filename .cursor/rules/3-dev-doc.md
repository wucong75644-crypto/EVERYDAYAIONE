## 必须做 ✅

### 1. 检查前置
- 需求清单是否明确？如缺失，提醒回到需求阶段。
- A级任务是否已执行方案评审（`/everydayai-evaluate`）？如已执行，将评审共识作为设计输入；如未执行，提醒用户考虑是否需要先评审。
- UI文档是否完成（如涉及前端）？如缺失，提醒完成UI设计。

### 2. 项目上下文加载（设计前必经，不可跳过）

AI 必须自动读取以下文件，理解项目当前架构后才能开始设计：

**必读文件**：
- `docs/document/TECH_ARCHITECTURE.md` — 系统架构全貌
- `docs/PROJECT_OVERVIEW.md` — 模块划分和职责边界
- `docs/CURRENT_ISSUES.md` — 避免踩已知坑
- `docs/FUNCTION_INDEX.md`（搜索相关函数）— 已有实现，避免重复

**必查**：
- 相关模块的现有 service 数量和文件行数（是否已膨胀需要拆分）
- 相关数据表的现有字段和索引（避免重复或冲突）
- 现有类似功能的实现方式（保持一致性，`Grep` 搜索关键词）
- 现有目录结构（新文件放哪里、是否需要新建子目录）

**产出**（必须完整包含以下 4 点）：
- **架构现状**：[3-5句话总结当前系统在相关领域的架构]
- **可复用模块**：[已有哪些可以直接用的]
- **设计约束**：[必须遵守的现有模式和约定]
- **潜在冲突**：[已知问题或与本次需求可能冲突的点]

**原则**：所有设计决策必须基于此步的实际代码和架构理解，禁止凭经验假设。

**红线**：项目上下文必须完整包含以上 4 点才能继续后续步骤，缺任何一点禁止进入第 6 步架构影响评估。如果某一点确实不适用（如无潜在冲突），必须显式标注"无"而非跳过。

### 3. 深度代码阅读
- **阅读范围**：涉及修改/新增的文件 + 上下游调用链 + 相关 Store/Hook/工具函数
- **理解目标**：数据流向、状态管理方式、命名约定、错误处理模式、现有复用逻辑
- **产出**：现有架构理解（3-5句），可复用模块，设计约束

### 4. 边界与极限情况分析
结合需求和现有代码，**主动穷举**以下场景并逐一给出处理策略：
- 空值/空列表/未初始化状态
- 网络异常/超时/重试
- 并发操作/竞态条件
- 数据量极大（分页、虚拟滚动）
- 用户快速连续操作（防抖/节流/取消）
- 权限不足/未登录/token过期
- 回退/撤销/中途取消

**产出**：边界场景清单表（场景 | 处理策略 | 涉及模块），纳入后续设计

### 5. 连锁影响评估（独立思维过程）
在理解代码和边界场景后，追踪每个计划改动点的完整调用链，找出所有必须同步修改的代码：
- 函数签名变更（入参/返回值）→ 所有调用方
- 类型/接口定义变更 → 所有引用该类型的文件
- Store 状态结构变更 → 所有消费该状态的组件
- API 请求/响应格式变更 → 前端调用方 + 后端处理方
- 数据库字段变更 → ORM模型 + 查询语句 + 序列化逻辑
- 组件 Props 变更 → 所有父组件传参处

**产出**：**连锁修改清单**（改动点 | 影响文件 | 必须同步修改内容），全部纳入后续任务拆分

### 6. 架构影响评估（A级任务必做）

从以下维度评估本次变更对系统架构的影响：

| 维度 | 评估内容 |
|------|---------|
| **模块边界** | 是否需要新增模块？是否跨越了现有模块边界？职责划分清晰吗？ |
| **数据流向** | 新增的数据流是否与现有模式一致？有没有引入环形依赖？ |
| **扩展性** | 用户量/数据量 10x 时还能撑住吗？瓶颈在哪？ |
| **耦合度** | 与多少个现有模块产生耦合？耦合是否必要？能否通过接口解耦？ |
| **一致性** | 新方案与现有代码的模式（错误处理、认证、日志、缓存）是否一致？ |
| **可观测性** | 上线后怎么知道它工作正常？需要什么日志/指标/告警？ |
| **可回滚性** | 部署出问题能回滚吗？数据库迁移是否可逆？ |

**产出**：

| 维度 | 评估 | 风险等级 | 应对措施 |
|------|------|---------|---------|
