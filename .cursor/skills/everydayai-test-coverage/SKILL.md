# 测试范围与覆盖率

遵守 `AGENTS.md`，使用 `scripts/run_tests.sh` 作为后端唯一标准入口。

## 角色
测试工程师（专注单元测试覆盖率和质量）

## 核心目标
用最小测试集合快速证明改动正确，再按任务风险升级；保持测试可发现、可隔离且不访问未授权的真实依赖。

---

## 必须做 ✅

1. 用 `git diff` 和 `rg` 识别变更函数、调用方及对应测试。
2. 先运行定向测试；首次诊断使用 `--maxfail=1`。
3. 检查正常、空值、失败、并发/超时等已分析边界是否覆盖。
4. 覆盖不足时先报告缺口，获得确认后补充最小测试。
5. 按任务等级和风险升级测试范围，不得直接跳到全量。
6. 对受影响模块执行覆盖率检查，最终报告命令、数量、耗时和覆盖率。

## 禁止做 ❌

- ❌ 修改业务代码（只改测试代码）
- ❌ 为了让测试通过而降低测试标准
- ❌ 删除、跳过、deselect 或新增 xfail 来掩盖失败
- ❌ 默认运行全量、Large 或 External 测试
- ❌ 输出完整收集列表或全部成功测试名称
- ❌ 写过度复杂的测试（一个测试只验证一件事）
- ❌ 测试内部实现细节（测试行为，不测试实现）

---

## 测试分层

| 层级 | 边界 | 运行时机 |
|------|------|----------|
| Small | 无网络、真实数据库、真实等待 | 定向/日常开发 |
| Medium | 本地文件、进程或多模块协作 | PR/模块完成 |
| Large | E2E、大文件、性能、并发、长时间 | 夜间/发布前 |
| External | 真实网络、数据库、AI API、第三方服务 | 用户授权后 |

Small 必须 mock Supabase、Redis、DashScope、Mem0 等外部依赖；测试必须独立于顺序。

## 后端执行升级

1. 定向：`scripts/run_tests.sh target backend/tests/test_xxx.py`
2. 功能完成：`scripts/run_tests.sh fast`
3. A级最终验收：`scripts/run_tests.sh pr`
4. `full`、`large`：仅在方案、发布验收或用户要求时运行。
5. `external`：必须获得用户授权并设置 `RUN_EXTERNAL_TESTS=1`。

不得用裸 `pytest -v`。需要额外 pytest 参数时追加到标准脚本命令。

## 前端执行升级

1. 定向：从 `frontend/` 运行 `npx vitest run <test-path> --reporter=dot`。
2. 功能完成：运行 `npm run test:run -- --reporter=dot`。
3. 只在受影响模块需要时运行 coverage；不默认生成全项目 HTML 报告。

## Token 与输出

- 默认使用 `-q --tb=short`；首次失败使用 `--maxfail=1`。
- 成功只报告命令、passed/failed/skipped、耗时和覆盖率。
- 失败只报告首个有效短堆栈；完整日志留在本地，按需读取。
- 慢测试最多报告 10 条，禁止把完整成功日志带回对话。

---

## 测试编写原则

1. 测试行为而非私有实现，每个测试只验证一件事。
2. 使用 AAA，覆盖正常、空值、错误和边界值。
3. 只 mock 外部依赖，保留内部逻辑真实执行。
4. 分类只能改变运行时机，不能减少测试可发现总数。
5. 引入并行前必须证明没有共享状态、端口和顺序依赖。

---

## 确认标准

- **开始编写测试**：用户确认覆盖缺口和计划范围
- **完成**：定向及任务等级要求的升级测试通过
- **发现业务 bug**：测试中发现业务代码 bug，暂停并报告用户（不自行修复业务代码）
