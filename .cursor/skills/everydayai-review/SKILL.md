# EverydayAI Code Review

对本次变更执行四层审查：多角色快速审查 → 方案完整性 → 代码质量 → 运行验证。

## 准备工作

1. 运行 `git diff --name-only HEAD` 获取变更文件列表
2. 运行 `git diff` 获取完整 diff
3. 阅读每个变更文件的完整内容（不只看 diff，要理解上下文）
4. 如有关联的调用方或被调用方，一并阅读
5. 如有相关测试文件，一并阅读

---

## 第零层：多角色交叉审查

以 5 个不同角色的视角，对本次变更做出独立判断。**每个角色只输出发现的问题，没有问题的角色完全跳过不输出。**

### 架构师视角
- 这个变更是否符合现有架构模式？（对照 `docs/document/TECH_ARCHITECTURE.md`）
- 模块边界是否清晰？有没有引入不必要的耦合？
- 如果系统规模扩大 10 倍，这个设计撑得住吗？
- 新增的模块/文件放置位置是否合理？

### 安全工程师视角
- 有没有新增的攻击面？（新接口、新用户输入、新文件处理）
- 认证和授权是否完整？受保护路由是否都有 auth 检查？
- 敏感数据处理是否合规？（日志中有没有打印 token/密码/PII）
- 外部输入是否做了校验和转义？

### 性能工程师视角
- 有没有引入慢查询或 N+1 问题？
- 前端有没有不必要的重渲染？（useEffect 依赖、状态结构）
- 异步操作是否合理？有没有阻塞主线程/事件循环的风险？
- 大数据量场景下是否有分页/虚拟滚动/懒加载？

### 可靠性工程师视角（SRE）
- 这个功能挂了会影响什么？需要降级方案吗？
- 日志够不够定位问题？关键操作有没有记录？（loguru + 业务上下文）
- 外部依赖（AI API、OSS、Redis、Supabase）超时或不可用时怎么办？
- 有没有重试/熔断/超时控制？

### 接手者视角
- 半年后的新人能看懂这段代码吗？
- 命名是否自解释？关键决策有没有注释说明"为什么"？
- 修改这块代码时，会不会意外影响其他地方？影响范围是否可预测？
- 数据流向是否清晰可追踪？

### 输出规则
- **只输出有问题的角色**，没有发现问题的角色完全跳过，不输出"未发现问题"
- 每条发现格式：`[角色名] 问题描述 | 风险等级（致命/需关注/建议）| 具体建议`
- 不同角色之间如果有矛盾观点（如安全要求 vs 性能要求），必须明确指出冲突并给出权衡建议

### 与后续层次的关系
- 第零层发现的问题，在第一层和第二层中深入分析
- 如果任何角色提出"致命"级问题，直接标记为 ❌ BLOCK

## 第一层：方案审查

从多个角度审视本次变更的完整性和合理性。以下维度作为思考起点，而非逐项打勾的清单——重点是发现问题，不是证明没问题。

### 设计合理性审查

站在系统之外俯瞰全貌：

- 这个方案的设计思路本身合不合理？有没有更简单直接的实现方式？
- 改了多个文件，这些文件之间的协作关系对不对？有没有职责混乱或边界模糊？
- 这次变更在整个系统架构中处于什么位置？和周边模块的边界划得清吗？
- 功能做完了，但解决的是不是真正的问题？有没有治标不治本？
- 有没有过度设计？简单的事情搞复杂了？

### 逻辑与 Bug 审查

逐文件检查变更代码的逻辑正确性：

- **条件分支**：if/else 是否覆盖所有情况？有没有遗漏的分支或互斥条件写错？
- **边界值**：空值、零值、负数、超长字符串、空数组、undefined——处理了吗？
- **异步时序**：多个异步操作之间有没有隐含的先后依赖？顺序颠倒会不会出错？
- **类型转换**：string/number 混用、null/undefined 混用、隐式类型转换是否安全？
- **循环/递归**：终止条件是否正确？有没有死循环或无限递归的可能？
- **错误传播**：catch 之后的处理对不对？有没有吞掉错误导致静默失败？上层调用方能否正确感知错误？
- **业务逻辑**：计算公式对不对？状态流转是否合法？权限判断是否正确？

### 换位思考

- **作为用户**：操作流程自然吗？会不会在某步困惑或卡住？出错时有没有明确反馈？
- **作为攻击者**：怎样绕过限制？输入什么会让系统出错或泄露信息？
- **作为接手的开发者**：半年后能看懂吗？改一处会不会连锁崩溃？
- **作为运维**：上线后日志/监控/报警会不会出现噪音？出问题时能否快速定位？

### 反向验证

- 列出本次变更的核心假设，逐个反问：什么时候会不成立？
- 这次改动**没动但依赖了**的代码，行为是否真如假设？
- 已有数据、已有调用方，是否仍然正常？

### 全局影响

- 数据从产生到消费的完整链路，每一跳是否都正确传递？有没有断链？
- 前端发的东西和后端收的东西，字段名/类型/枚举值/可选必填是否对齐？
- 新增的状态（内存/缓存/localStorage/DB），何时创建、何时清除？有没有只写不清？
- 多入口场景（Web/API/WebSocket/定时任务/第三方回调），是否都覆盖了？
- 部署顺序有无依赖？中间态会不会报错？回滚是否安全？

### 输出要求

- 如果发现问题，用自然语言描述具体场景和风险
- 某个维度确认无风险则跳过，不需要输出"通过"
- **只输出有价值的发现**

## 第二层：代码审查

### 2.1 安全（CRITICAL — 必须修复）

只报告 >80% 确信的真实问题。按以下清单扫描变更文件：

- **硬编码凭证**：API key / 密码 / token / 连接串出现在源码中
- **注入漏洞**：SQL 字符串拼接、shell 命令拼接用户输入、不安全的 eval/exec
- **XSS**：未转义的用户输入渲染到 HTML/JSX
- **路径穿越**：用户可控的文件路径未做规范化校验
- **认证绕过**：受保护路由缺少 auth 检查
- **CSRF**：状态变更接口缺少 CSRF 防护
- **日志泄露**：日志中打印了密码、token、PII 等敏感信息
- **不安全反序列化**：pickle.loads / yaml.unsafe_load 处理用户输入

### 2.2 代码质量（HIGH）

- **文件 >500 行** → 需拆分
- **函数 >120 行** → 需重构
- **嵌套 >4 层** → 使用 early return 或提取函数
- **重复代码**（≥3行 × 2处）→ 提取公共函数
- **缺少错误处理** → 空 catch / 未处理的 Promise rejection
- **死代码** → 未使用的 import、注释掉的代码、unreachable branch
- **console.log / print** → 移除调试输出

### 2.3 框架专项（HIGH）

**React / TypeScript：**
- useEffect / useMemo / useCallback 依赖数组是否完整？
- 列表渲染是否使用稳定唯一 key（禁止 index as key 于可重排列表）？
- 是否存在 stale closure（事件处理器捕获了过期状态）？
- 异步请求是否有 cleanup（AbortController / 取消标志）？
- 类型定义是否使用了 `any`？能否收窄为具体类型？

**FastAPI / Python：**
- async 路由内是否调用了同步阻塞操作？
- Pydantic model 是否覆盖了请求校验？
- 可变默认参数（`def f(x=[])`）？
- bare except（`except: pass`）？
- 资源是否使用 context manager（`with` 语句）？

**数据库：**
- 是否存在 N+1 查询？
- 用户侧接口是否有 LIMIT 防止无界查询？
- 多步操作是否在事务中执行？

### 2.4 依赖审查（MEDIUM）

- 新增的依赖是否必要？有没有项目已有的依赖能替代？
- 依赖的维护状态如何？是否仍在积极维护？
- 依赖的安全记录（已知 CVE）

### 2.5 性能（MEDIUM）

- O(n²) 可优化为 O(n log n) 或 O(n) 的场景
- 缺少缓存的重复计算
- 未做懒加载的大资源
- 同步 I/O 阻塞异步上下文

### 2.6 规范（LOW）

- 魔法数字未提取为常量
- 公开函数缺少类型注解（Python）
- 命名不清晰（单字母变量用于非循环场景）

## 第三层：运行验证

根据变更文件类型，执行对应的自动化检查：

**如果有 Python 变更：**
```bash
source backend/venv/bin/activate && python -m pytest backend/tests/ -q --tb=short
```

**如果有 TypeScript/React 变更：**
```bash
cd frontend && npx tsc --noEmit
cd frontend && npx vitest run
```

如果测试或类型检查失败，在报告中标记为 HIGH。

## 输出格式

### 第零层结果（多角色审查）

只输出有发现的角色（没问题的角色不出现）：
```
[架构师] xxx | 需关注 | 建议xxx
[安全工程师] xxx | 致命 | 必须xxx
```

如果所有角色均无发现，输出："多角色审查未发现问题"。

### 第一层结果

用自然语言按发现的问题逐条描述，每条包含：
- 问题场景
- 风险等级（致命 / 需关注 / 建议）
- 具体影响

无发现则输出："方案审查未发现问题"。

### 第二层结果

每条 issue 格式：
```
[SEVERITY] 问题标题
File: path/to/file.ts:42
Issue: 具体描述
Fix: 修复建议（含代码示例）
```

相似问题合并为一条，注明出现次数。

### 汇总

```
## Review Summary

| 严重度 | 数量 | 状态 |
|--------|------|------|
| CRITICAL | 0 | ✅ |
| HIGH | 0 | ✅ |
| MEDIUM | 0 | — |
| LOW | 0 | — |
| 多角色风险 | 0 | ✅ |
| 方案风险 | 0 | ✅ |

Verdict: ✅ APPROVE / ⚠️ WARNING / ❌ BLOCK
```

### 判定标准

| 结果 | 条件 |
|------|------|
| ✅ APPROVE | 无 CRITICAL / HIGH，多角色审查和方案均无致命风险 |
| ⚠️ WARNING | 仅 HIGH 或"需关注"级风险（可审慎提交） |
| ❌ BLOCK | 存在 CRITICAL 或任何层面的致命风险 — 必须修复 |

---

## 审查纪律

- 只报告 >80% 确信的真实问题，不报风格偏好
- 未改动的代码不审查（CRITICAL 安全问题除外）
- 相似问题合并为一条，注明数量
- 每条代码 issue 必须有 File:line + Fix，禁止只指问题不给方案
- 方案层面的风险必须描述具体场景，禁止泛泛而谈
- 第零层鼓励多角度发散，第一层深入分析，第二层严格按清单执行，不要混淆
