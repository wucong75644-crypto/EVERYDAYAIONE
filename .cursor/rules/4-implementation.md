# 阶段4：开发执行

> **前置**：需求挖掘 + UI设计 + 技术设计
> **触发**：`@4-implementation` 或用户说"开始开发""写代码"
> **底层规则**：必须遵守基础开发规则所有条款（特别是任务分级）

## 角色
执行开发者 / 程序员

## 核心目标
严格按设计文档实现代码，不自作主张，分模块逐步完成，**每步验证、全程质量把控**

---

## 任务分级
参考 AGENTS.md §2.1。AI 在开始任务前必须告知用户任务级别。

---

## 必须做 ✅
1. **检查前置**：设计文档是否完整（数据库、API、文件清单）？如缺失，提醒回到设计阶段
2. **范围锁定**：只实现设计文档中的内容、只修改【文件清单】中的文件、禁止修改范围外代码
3. **分步实现**：一次只完成一个任务（如"任务1.1：创建数据库表"），每完成一个报告进度，等待确认后再开始下一个
4. **代码质量**：遵守基础规则硬性指标（500/120/15/4）、添加注释、添加错误处理（try-except+loguru）
5. **文档同步**：新增/修改函数必须同步更新 `FUNCTION_INDEX.md`、修复问题必须同步更新 `CURRENT_ISSUES.md`
6. **发现问题立即暂停**：设计文档有问题或遗漏时，立即暂停并报告用户，不得擅自修改设计

## 禁止做 ❌
- ❌ 自作主张添加设计文档没有的功能
- ❌ 自作主张修改数据库结构、API接口、UI组件
- ❌ 一次性输出所有代码
- ❌ 重构或优化设计文档没提到的东西
- ❌ 使用TODO/workaround/占位符

## 开发全流程（三阶段）

### 阶段一：开发前准备（每次开发必经）

#### 1.1 实现级代码精读
设计阶段已完成架构级阅读，开发阶段需进一步精读**实现细节**：
- **精读范围**：当前子任务涉及的文件（逐行阅读），重点关注：
  - 函数内部实现逻辑、参数校验方式、返回值格式
  - 现有错误处理 pattern（try-catch 风格、错误码定义）
  - 已有 TypeScript 类型定义、接口约定
  - 测试文件的 mock 方式和断言风格
- **对比设计**：确认设计文档中的方案与当前代码**无冲突**，如有变动（他人提交、依赖升级）立即报告
- **产出格式**：
  - `文件:函数名(行号)` → 改动说明
  - 设计对比：✓ 无冲突 / ⚠ 冲突：xxx

#### 1.2 调用方影响扫描
修改任何函数/接口前，必须先搜索所有调用方：
- 使用 Grep 搜索函数名、API路径、组件名
- 列出所有调用方及影响范围
- 确认改动不会破坏已有调用
- **与设计阶段交叉验证**：确认调用方范围与设计文档一致，如有新增调用方须补充影响评估
- **产出格式**：`函数名` → 调用方：fileA:行号, fileB:行号 → 兼容：✓/⚠

#### 1.3 基线测试
开发前先跑一遍全量现有测试，记录基线状态：
- **后端**：`source backend/venv/bin/activate && python -m pytest backend/tests/ -q --tb=short`
- **前端**：`cd frontend && npx vitest run`
- **记录**：通过数/失败数，作为开发后对比基准
- **原则**：如果基线测试已有失败，记录下来，开发过程中不能增加新的失败

### 阶段二：逐步开发（循环执行）

每完成一个子任务，必须依次执行以下检查，**全部通过才能进入下一个子任务**：

#### 2.1 实现代码
- 严格按设计文档实现
- 确保向后兼容，不破坏现有功能

#### 2.2 单元测试（门禁）
每步实现后立即执行：
- **跑已有测试**：确认没有引入回归
- **补齐新测试**：为新增/修改的函数编写单元测试
- **测试覆盖**：核心业务逻辑覆盖率≥80%
- **必须全部通过**：任何测试失败必须立即修复，不得跳过

```
测试命令：
后端：source backend/venv/bin/activate && python -m pytest backend/tests/ -q --tb=short
前端：cd frontend && npx vitest run
```

#### 2.3 TypeScript 类型检查（前端改动时）
前端代码改动后必须执行：
```
cd frontend && npx tsc --noEmit
```
- 有类型错误必须立即修复

#### 2.4 代码质量门禁
对本步修改的每个文件：`wc -l` 实测行数 → 500/120/15/4 全过才继续（详见 AGENTS.md §三）

#### 2.5 Git Diff 最小化审查
每步完成后检查改动范围：
- 确认 diff 中没有超出当前子任务范围的修改
- 确认没有"顺手改"的无关代码
- 确认没有遗留调试代码（console.log、print等）

#### 2.6 步骤完成报告
```
✅ 任务X.X完成

**质量门禁**（必填，附 wc -l 证据）：
| 文件 | 行数 | 最长函数 | 状态 |
|-----|------|---------|------|
| xxx.py | 180行 | 45行 | ✓ |

**改动文件**：
| 文件路径 | 操作 | 行数 | 说明 |
|---------|------|------|------|
| src/api/xxx.py | 新建 | 120 | xxx API |

**测试结果**：
- 后端：XX passed / XX total（与基线一致）
- 前端：XX passed / XX total（与基线一致）
- 新增测试：X个

**类型检查**：✓ 通过（如涉及前端）

---
请确认是否继续下一个任务？
```

**循环规则**：2.1 → 2.2 → 2.3 → 2.4 → 2.5 → 2.6 → 确认 → 下一个子任务

### 阶段三：开发完成验收（所有子任务完成后）

所有子任务完成后，AI 必须自主执行以下全套验收，**全部通过后才通知用户验收**：

#### 3.1 全量回归测试
重新跑全量测试，与基线对比：
- 后端全量测试
- 前端全量测试
- **对比基线**：不能有新增失败

#### 3.2 代码体检
对所有新增/修改文件执行质量门禁：500/120/15/4 + 重复检查 + 单一职责（详见 AGENTS.md §三），逐文件附 wc -l 证据。

#### 3.3 安全扫描
- 无硬编码密钥/API Key
- 无 SQL 注入风险
- 无 XSS 风险
- 敏感信息从环境变量读取

#### 3.4 构建验证
确保项目能正常构建：
- **后端**：无 import 错误，服务能正常启动
- **前端**：`cd frontend && npm run build` 构建通过

#### 3.5 AI 全链路功能测试
AI 自主进行端到端功能验证：
- 启动后端服务，验证 API 接口可正常调用
- 验证核心功能链路（根据设计文档中的功能点逐一验证）
- 验证边界场景（空值、异常输入、并发等）
- 记录测试结果

#### 3.6 验收报告（全部通过后才输出）
