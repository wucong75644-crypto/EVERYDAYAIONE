# 方案评审：多角色辩论

> **触发**：`/everydayai-evaluate` 或用户说"评审方案""讨论一下""这样行不行"
> **定位**：对关键技术决策进行多角色深度辩论，形成经过多视角验证的推荐方案
> **适用场景**：技术选型、架构变更、新增外部依赖、重构方向、成本敏感决策
> **底层规则**：必须遵守基础开发规则所有条款

## 核心原则
这不是检查清单，是**模拟专家会议**。每个角色有自己的立场和关注点，角色之间会互相质疑和挑战，最终形成经过碰撞的共识。

---

## 评审角色（根据议题自动选择 3-5 个最相关的角色）

### 可用角色池

| 角色 | 关注点 | 适用场景 |
|------|--------|---------|
| **系统架构师** | 模块边界、扩展性、数据流、耦合度、设计模式一致性 | 所有A级任务 |
| **安全工程师** | 攻击面、认证授权、数据泄露、OWASP、合规 | 涉及用户数据/新接口/认证 |
| **性能工程师** | 查询效率、渲染性能、缓存策略、异步设计、大数据量 | 涉及数据库/列表/高频操作 |
| **成本控制专家** | API调用费用、云资源成本、存储成本、维护人力成本 | 涉及外部API/云服务/新依赖 |
| **运维/SRE专家** | 部署复杂度、监控、故障恢复、回滚、可观测性 | 涉及部署变更/新服务/外部依赖 |
| **产品视角** | 用户体验、业务价值、优先级、功能边界 | 涉及用户交互/功能取舍 |
| **接手者视角** | 代码可读性、文档完整性、维护难度、学习曲线 | 复杂度高的方案 |

#### 仲裁角色（不参与常规三轮，仅在分歧时自动介入）

| 角色 | 关注点 | 触发条件 |
|------|--------|---------|
| **行业标准专家** | 行业最佳实践、RFC/规范、主流开源方案的通行做法、权威技术文档的推荐模式 | 第二轮交叉质疑产生明确分歧时，第三轮自动介入裁决 |

角色选择原则：常规角色根据议题自动选择 3-5 个最相关的，避免与议题无关的角色发言产生噪音。行业标准专家不参与选择，仅在出现分歧时自动触发。

## 项目上下文绑定（评审前必经）

评审开始前，AI 必须先加载项目上下文：
- 阅读 `docs/document/TECH_ARCHITECTURE.md` — 系统架构全貌
- 阅读 `docs/PROJECT_OVERVIEW.md` — 模块划分和职责边界
- 搜索相关现有实现（Grep 关键词）

**所有角色的发言必须基于项目实际情况**，禁止脱离项目空谈。例如：
- 架构师必须参照现有架构模式发言（FastAPI + React 单体、Supabase PostgreSQL）
- 成本专家必须参照项目实际使用的云服务和API（Aliyun OSS、Gemini API、Upstash Redis）
- 性能工程师必须参照项目实际的数据量级和并发量
- 运维专家必须参照项目实际的部署方式（systemd + Nginx + deploy.sh）

## 评审流程（三轮递进）

### 第一轮：独立分析（每角色 3-5 句话）

每个角色从自己的专业视角，对方案做出独立判断：
- 这个方案从我的角度看，**最大的优点**是什么？
- **最大的风险/问题**是什么？
- 如果必须改一个地方，我会改什么？

**输出格式**：
```
### 第一轮：独立分析

**[架构师]**
优点：xxx
风险：xxx
建议改动：xxx

**[安全工程师]**
优点：xxx
风险：xxx
建议改动：xxx

...
```

### 第二轮：交叉质疑（角色之间互相挑战）

每个角色针对**其他角色的分析**提出质疑或补充：
- "架构师说要拆微服务，但从成本角度这样做值吗？"
- "安全工程师要求加密所有字段，但性能开销评估了吗？"
- "成本专家说用免费方案，但运维复杂度谁来承担？"

**要求**：
- 每个角色至少对一个其他角色的观点提出质疑
- 质疑必须具体（不能说"我不同意"，要说"你说xxx，但在xxx场景下会导致xxx"）
- 被质疑的角色必须回应

**输出格式**：
```
### 第二轮：交叉质疑

**[成本专家 → 架构师]**
质疑：你建议新增独立服务，但目前团队就一个人，维护成本...
架构师回应：同意，可以先用模块拆分代替服务拆分，保持单体部署...

**[性能工程师 → 安全工程师]**
质疑：全字段加密在列表查询场景下会导致...
安全工程师回应：敏感字段加密即可，非敏感字段不需要...

...
```

### 第三轮：形成共识 + 分歧仲裁（收敛结论）

基于前两轮的讨论，先识别共识与分歧，再对分歧点引入行业标准专家裁决：

**分歧仲裁规则**：
- 第二轮交叉质疑中产生的每一个明确分歧，行业标准专家必须介入
- 引用具体的行业标准、RFC、主流开源项目做法、权威技术文档（如 AWS Well-Architected、OWASP、12-Factor、PostgreSQL 官方文档等）
- 给出裁决建议：基于行业标准推荐哪一方 + 理由
- 必须评估适用性：本项目是否有特殊情况需要偏离标准
- 如果行业内本身没有共识（两种做法都是主流），必须明确说明，交用户决策
- 无分歧时跳过仲裁环节，直接输出共识

**输出格式**：
```
### 第三轮：共识结论

#### 推荐方案
[综合各角色意见 + 行业标准仲裁后的推荐方案，1-3句话]

#### 各角色达成的共识
1. [共识点1]：所有角色同意xxx
2. [共识点2]：xxx

#### 分歧仲裁（行业标准专家）
**分歧1**：[架构师]认为xxx，[成本专家]认为xxx
- 行业标准：[引用具体标准/RFC/主流做法]
- 裁决建议：推荐xxx，因为xxx
- 适用性：本项目xxx，[符合标准/需偏离标准，原因是xxx]

**分歧2**：...
（如无分歧，此节省略）

#### 仍需用户决策的问题
1. [问题1]：行业内两种做法都是主流，xxx vs xxx → **需要用户根据业务优先级判断**
（如行业标准专家已裁决所有分歧，此节标注"无"）

#### 必须满足的约束（底线）
1. [安全底线]：xxx 不可妥协
2. [性能底线]：xxx

#### 风险接受声明
- 选择此方案意味着接受：xxx（由xxx角色指出）
- 缓解措施：xxx
```

## 与其他技能的关系

```
requirement（需求确认）
    ↓
evaluate（关键决策辩论）← 本技能
    ↓
tech-design（将 evaluate 的共识落地为具体设计文档）
    ↓
implementation → review → git-push → deploy
```

**触发时机**：
- **在 requirement 之后、tech-design 之前**：对整体方案方向进行辩论（最常见）
- **在 tech-design 过程中**：遇到关键分叉点（技术选型、架构变更）时临时触发
- **独立使用**：用户主动说"帮我讨论一下这个方案"

**与 tech-design 的区别**：
- evaluate = **讨论"做不做、怎么做的方向"**（发散、辩论、决策）
- tech-design = **输出"具体怎么做的文档"**（收敛、落地、任务拆分）

**与 review 第零层的区别**：
- evaluate = **事前**，方案还没确定，多角色帮你选方向
- review 第零层 = **事后**，代码已经写完，多角色帮你找问题

---

## 禁止做 ❌
- ❌ 所有角色意见一致（如果完全没有分歧，说明讨论不够深入，强制找出至少一个值得权衡的点）
- ❌ 角色发言脱离项目实际（如讨论微服务但项目是单体、讨论k8s但项目用systemd）
- ❌ 跳过第二轮交叉质疑（这是最有价值的环节）
- ❌ 第三轮模糊共识（"大家都觉得还行"不算共识，必须有具体结论）
- ❌ 写任何代码或设计数据库表（这是 tech-design 的职责）
