## 一、EVERYDAYAI 开发流程技能（斜杠命令）

### 主流程（按顺序使用）

A级任务走完整流程，B级任务可跳过标注"可选"的步骤。

| 序号 | 命令 | 触发条件 | 功能 | B级可跳过 |
|------|------|---------|------|----------|
| 1 | `/everydayai-requirement` | 说"我想要...""加一个功能" | **需求挖掘**：自动任务分级(A/B/C) → 结构化提问 → AI自动读取项目文档评估现状 → 非功能性需求(A级) → 输出需求清单+成功标准 | 第3步非功能性需求标注"待评估"即可 |
| 2 | `/everydayai-evaluate` | 说"评审方案""讨论一下""这样行不行" | **方案评审（多角色辩论）**：AI自动选择3-5个专家角色 → 三轮递进（独立分析→交叉质疑→形成共识）→ 输出推荐方案+分歧点+底线约束。所有角色基于项目实际情况发言 | 整步跳过（B级无需辩论） |
| 3 | `/everydayai-ui-design` | 说"界面设计""页面布局" | **UI设计**：页面结构、交互流程、状态设计、组件清单 | 纯后端功能整步跳过 |
| 4 | `/everydayai-tech-design` | 说"技术方案""API设计" | **技术设计**：强制项目上下文加载(4点必填) → 深度代码阅读 → 边界场景 → 连锁影响评估 → 架构影响评估(A级) → 方案对比(A级) → 数据库/API/任务拆分 → 回滚策略(A级) | 跳过架构评估、方案对比、回滚策略 |
| 5 | `/everydayai-implementation` | 说"开始开发""写代码" | **开发执行**：代码精读 → 基线测试 → 逐步实现(每步：代码→单测→类型检查→质量门禁) → 全量验收 | - |
| 6 | `/everydayai-review` | 说"审查""review"，或开发完成后 | **四层审查**：①多角色交叉审查(架构师/安全/性能/SRE/接手者，只输出有问题的角色) ②方案审查(设计合理性/逻辑Bug/换位思考/反向验证/全局影响) ③代码审查(安全/质量/框架/依赖/性能) ④运行验证(pytest/tsc/vitest) → Verdict判定 | - |
| 7 | `/everydayai-git-push` | 说"推送代码""push" | **Git推送**：检查状态 → 生成commit message → 提交推送 | - |
| 8 | `/everydayai-deploy` | 说"部署""deploy""上线" | **部署生产**：确认已推送 → 选择范围(前端/后端/全量) → 执行deploy.sh → 验证 | - |

### 辅助技能（随时调用）

| 命令 | 触发条件 | 功能 |
|------|---------|------|
| `/everydayai-testing` | 说"有bug""报错""改一下" | **Bug修复**：分析问题 → 提出方案 → 确认后最小改动修复 → 回归验证 |
| `/everydayai-test-coverage` | 说"检查测试""补充测试" | **测试补充**：扫描变更文件 → 识别缺失测试 → 补充并运行通过 |
| `/everydayai-restart` | 说"重启服务""restart" | **重启服务**：杀死旧进程 → 释放端口(8000/3000) → 重启前后端 → 验证 |

## 二、任务分级速查

每次开始新任务时，AI 会自动判断任务等级：

| 等级 | 判断标准 | 流程深度 |
|------|---------|---------|
| **A级** | ≥3文件 / 核心逻辑 / 新功能模块 / 架构变更 | 完整流程：需求 → UI(可选) → 技术设计(含架构评估+方案对比) → 开发 → 审查 → 推送 |
| **B级** | 单文件、<50行 / 加字段、改接口 | 精简流程：需求(精简) → 技术设计(精简) → 开发 → 审查 → 推送 |
| **C级** | 依赖变更 | 最小流程：影响分析 → 执行 |

## 三、V2.0 新增能力（本次升级重点）

### 1. 多角色辩论式方案评审（evaluate，A级推荐）
- **核心能力**：模拟专家会议，多角色深度辩论技术方案
- 7个常规角色：架构师、安全工程师、性能工程师、成本控制、运维/SRE、产品视角、接手者
- 1个仲裁角色：**行业标准专家**（不参与常规三轮，仅在分歧时自动介入，引用行业标准/RFC/主流方案裁决）
- 根据议题自动选择 3-5 个最相关常规角色
- **三轮递进 + 仲裁**：独立分析 → 交叉质疑（角色之间互相挑战）→ 形成共识 + 分歧仲裁（行业标准专家裁决）
- 所有角色基于项目实际情况发言（自动读取架构文档），禁止脱离项目空谈
- 产出：推荐方案 + 共识点 + 分歧仲裁（行业标准专家）+ 仍需用户决策的问题 + 底线约束

### 2. 项目现状自动评估（requirement + tech-design）
- AI 会**自动阅读** `PROJECT_OVERVIEW.md`、`CURRENT_ISSUES.md`、`TECH_ARCHITECTURE.md`
- 自动搜索现有代码中是否已有类似功能
- 所有设计决策基于实际代码，不凭经验假设

### 3. 架构影响评估（tech-design，A级必做）
- 7 个维度评估：模块边界、数据流向、扩展性、耦合度、一致性、可观测性、可回滚性
- **红线机制**：发现高风险架构问题时自动暂停，输出替代方案，等待用户决策

### 4. 方案对比（tech-design，A级必做）
- 至少 2 个方案从侵入性、复杂度、性能、风险等维度对比
- 用户确认方案后才继续设计

### 5. 多角色交叉审查（review 第零层）
- 5 个角色独立审查：架构师、安全工程师、性能工程师、SRE、接手者
- **只输出有问题的角色**，无问题的跳过不输出，减少噪音
- 角色之间矛盾观点会明确指出并给出权衡建议

### 6. 非功能性需求（requirement，A级必做）
- 性能、安全、可用性、可观测性四个维度
- B级任务标注"待评估"，不强制展开

## 四、关键红线（AI 不会违反）

| 红线 | 位置 | 说明 |
|------|------|------|
| 项目上下文 4 点必填 | tech-design 第2步 | 架构现状/可复用模块/设计约束/潜在冲突，缺一不可 |
| 架构高风险暂停 | tech-design 第6步 | 发现高风险自动暂停，不会带病设计 |
| 质量门禁 500/120/15/4 | implementation + review | 文件>500行/函数>120行/复杂度>15/嵌套>4层 |
| 审查 80% 确信 | review | 只报告高确信的真实问题，不报风格偏好 |
