# TECH: Agent 计划模式 v3 — 主 Agent 通用能力

> 版本：v3.0 | 日期：2026-04-25 | 状态：方案讨论

## 1. 问题与修正

### 1.1 v2.0 的错误

v2.0 把计划模式做成了 **erp_agent 的内部能力**（L2 短路返回 plan），但：
- L1 提示词教主 Agent "自己拆步执行"，导致主 Agent 绕过 L2，erp_agent 的计划模式永远不触发
- 计划模式只覆盖 ERP 查询，不覆盖跨工具场景（erp_agent → code_execute → generate_image）
- 主 Agent 是 plan 的搬运工，不是 plan 的主人

### 1.2 v3.0 的修正

**计划模式是主 Agent 的通用能力，覆盖所有工具，不只是 erp_agent。**

参考 Claude Code 的实现：计划模式是纯提示词驱动，不改循环架构。核心是让 LLM 在工具循环中自然地走过**探索→规划→确认→执行**四个阶段。

---

## 2. 核心设计：两种工作模式

### 2.1 判断标准（工具无关）

主 Agent 收到用户请求后做一个自检：

> **"完成这个任务，我是否需要先拿到某一步的结果，才能知道下一步该怎么做？"**

| 判断结果 | 模式 | 行为 |
|---------|------|------|
| 否——所有步骤的参数我现在都清楚 | **直接模式** | 自己拆步执行，不等确认 |
| 是——后续步骤的参数依赖前面步骤的产出 | **计划模式** | 探索→规划→展示→确认→分步执行 |

### 2.2 触发场景（覆盖所有工具）

| 场景 | 涉及工具 | 为什么需要计划 |
|------|---------|--------------|
| 查供应商商品→用编码查订单 | erp_agent → erp_agent | 第二步的 product_code 依赖第一步 |
| 导出订单→按平台分组→生成柱状图 | erp_agent → code_execute | 图表的数据源依赖导出结果 |
| 查竞品笔记→分析关键词→生成报告 | social_crawler → code_execute | 分析内容依赖爬取结果 |
| 查库存不足的商品→创建采购单 | erp_agent → erp_execute | 采购单的商品列表依赖库存查询 |
| 对比上月和本月销量→生成趋势图 | erp_agent ×2 → code_execute | 两次查询 + 计算 + 可视化 |

**不需要计划模式的场景（直接执行）：**

| 场景 | 原因 |
|------|------|
| 昨天订单汇总 | 单步，参数明确 |
| 退货率（订单+售后） | 两步独立并行，互不依赖 |
| 导出本周订单明细 | 单步导出 |

### 2.3 两种模式的完整流程

```
直接模式（主Agent自信能搞定）：
  用户 → 主Agent判断"我能一步搞定/参数都清楚"
       → 直接调工具 → 拿结果 → 呈现给用户

计划模式（需要先探索才能规划）：
  用户 → 主Agent判断"后续步骤依赖前面的结果"
       → 阶段1 探索：调 erp_agent 等工具了解任务结构
         （erp_agent 可能返回 status=plan 提供结构化分析）
       → 阶段2 规划：基于探索结果制定执行方案
       → 阶段3 展示：用自然语言向用户展示方案，停止工具调用
       → --- 等用户确认/调整 ---
       → 阶段4 执行：按方案逐步调工具，每步传入上一步的结果
       → 阶段5 汇总：所有步骤完成后输出完整结论
```

## 3. 实现方案：纯提示词驱动

### 3.1 设计原则（参考 Claude Code）

- **不改循环架构**：现有的 while 循环 + 15 轮预算足够覆盖探索→确认→执行全流程
- **纯提示词控制**：通过 TOOL_SYSTEM_PROMPT 教主 Agent 何时直接干、何时先规划
- **不限制工具**：不像 Claude Code 那样限制只读工具（我们的工具都是 API 调用，没有破坏性）
- **自然停顿**：展示方案后 LLM 自然停止（不再输出工具调用），用户下一条消息即为确认

### 3.2 erp_agent status=plan 的角色

erp_agent 已实现的 `_build_plan_result` / `status=plan` 机制**保留不变**，但角色重新定义：

| 角色 | 说明 |
|------|------|
| v2.0（旧） | 计划模式的**唯一触发器**——erp_agent 检测到复杂就返回 plan |
| v3.0（新） | 探索阶段的**信号之一**——erp_agent 分析查询结构并返回结构化信息，主 Agent 可能基于它规划，也可能基于自己的判断规划 |

主 Agent 进入计划模式有两条路径：
1. **主 Agent 自己判断**："这个任务后续步骤依赖前面的结果" → 直接进入计划模式
2. **erp_agent 返回 status=plan**：主 Agent 把完整查询交给 erp_agent，erp_agent 返回结构化分析 → 主 Agent 基于它进入计划模式

两条路径最终合流：主 Agent 展示方案 → 用户确认 → 分步执行。

### 3.3 TOOL_SYSTEM_PROMPT 完整替换

替换当前 `TOOL_SYSTEM_PROMPT`（chat_tools.py:106-156）为：

```python
TOOL_SYSTEM_PROMPT = """## 工具决策规则

### erp_agent — ERP 数据检索
用户问任何涉及订单/库存/采购/售后/发货/物流/商品/销量/统计的问题时调用。
task 原样传递用户的话，conversation_context 给最近相关的对话内容。
erp_agent 是 worker 进程，返回数据摘要或 staging 文件引用：
- 纯数字结论 → 直接向用户呈现，加上下文做适当解读
- 含 [文件已存入 staging] → 用户要导出时调 code_execute 读 staging 转 Excel
- 含 [关联计算提示] → 调 code_execute 按提示读多个 staging 文件关联计算
- 参数不足 → 用 ask_user 补充后重新调用
- 错误 → 告知用户并建议替代方案
- 能力约束（status=plan）→ erp_agent 分析出任务需要多步执行，返回了结构化的步骤和依赖关系。按下方"计划模式"处理
不要重复 erp_agent 的原始数据，基于它做呈现即可。

### search_knowledge — 知识库
业务规则、操作流程等非数据类问题。

### web_search — 互联网搜索
天气、新闻等实时信息。社交平台内容用 social_crawler。

### generate_image / generate_video
用户要求画图/生成视频时使用。

### code_execute — 代码执行
erp_agent 返回的 staging 文件需要转 Excel 或关联计算时使用，或处理用户上传的工作区文件。
读 Excel 用 engine='calamine'，写 Excel 用 engine='xlsxwriter'。大结果写文件不要 print(df)。

### 工作区文件
用 file_list 确认文件名，code_execute 读取处理。Excel/二进制不能用 file_read。

## 执行模式判断

收到用户请求后，先做一个自检：
**"完成这个任务，我是否需要先拿到某一步的结果，才能确定下一步该怎么做？"**

=== 直接模式（大部分场景）===
所有步骤的参数你现在都清楚 → 直接调工具执行，不等确认。
例：查昨天订单汇总、导出本周明细、退货率统计（各域独立并行）。

=== 计划模式（后续步骤依赖前面的产出）===
某一步的输入需要前面步骤的查询结果才能确定 → 进入计划模式。
例：查供应商商品再用编码查订单、导出数据再生成图表、查库存不足再创建采购单。

计划模式四阶段：
1. 探索：调 erp_agent（或其他工具）了解任务结构。
   erp_agent 可能返回 status=plan，包含结构化的步骤和依赖分析——利用它。
2. 规划：基于探索结果，制定执行方案（几步、每步做什么、数据如何传递）。
3. 展示并等待：用自然语言向用户展示方案。展示后停止，不要继续调工具。
   等用户确认（"可以""执行"）或调整（"时间改7天""不用分步"）。
4. 执行：用户确认后，按方案逐步调工具。每步拿到结果后传给下一步。
   全部完成后输出完整结论 + 推理逻辑。

## 对话交互

=== CRITICAL ===
- 业务数据问题必须调 erp_agent，禁止不查数据就回答
- 数据查询有歧义时调 ask_user 追问用户（猜错代价 > 多问一次）
- 需要追问时用 ask_user 工具，简洁语言 + 2-3 个选项引导选择
- 信息完整无歧义时直接执行，不要反复确认
- 你可以在一轮中调用多个工具，无依赖关系的工具调用应并行发起以提高效率。但如果某个调用依赖另一个的结果，必须顺序执行"""
```

| 位置 | 删除 | 新增 |
|------|------|------|

## 5. 与 Claude Code 的对比

| 维度 | Claude Code | 我们 |
|------|------------|------|
| 触发方式 | 手动（Shift+Tab） | 自动（LLM 自检判断） |
| 工具限制 | 计划阶段只允许只读工具 | 不限制（API 调用无破坏性） |
| 计划存储 | markdown 文件 | 对话文本（自然语言方案） |
| 确认机制 | EXIT_PLAN_MODE 工具 | 用户下一条消息 |
| 实现方式 | 纯提示词注入 | 纯提示词注入（一致） |
| 架构改动 | 无（提示词驱动） | 无（提示词驱动，一致） |

**关键相同点**：都是提示词驱动，不改循环架构，不加状态机。

**关键不同点**：Claude Code 是手动切换，我们是自动判断——因为业务用户不会手动切模式。自动判断的标准足够简单（"后续步骤是否依赖前面的产出"），LLM 可以可靠执行。

1. **场景1**：发送"查供应商纸制品01的商品，用编码查订单"
   - 预期：主 Agent 判断有依赖 → 调 erp_agent 探索 → erp_agent 返回 plan → 展示方案 → 用户确认 → 分步执行

2. **场景2**：发送"昨天订单汇总"
   - 预期：主 Agent 判断无依赖 → 直接调 erp_agent → 返回结果

3. **场景3**：发送"导出昨天订单按平台分组，生成柱状图"
   - 预期：主 Agent 判断有依赖（图表依赖数据）→ 展示方案 → 确认后 erp_agent 查数据 → code_execute 生成图表

4. **场景4**：发送"退货率多少"
   - 预期：主 Agent 判断无依赖（订单+售后独立并行）→ 直接执行

## 7. 风险评估

| 风险 | 严重度 | 缓解措施 |
|------|--------|---------|
| LLM 该计划时直接干了（false negative） | 中 | erp_agent L2 兜底返回 plan |
| LLM 不该计划时进入计划模式（false positive） | 低 | 用户说"直接查"即可跳过 |
| LLM 展示方案后不停止，继续调工具 | 中 | 提示词加粗"展示后停止" + 线上观察 |
